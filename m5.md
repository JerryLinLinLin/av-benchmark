# Milestone 5 ŌĆö Replace typeperf with system-wide disk I/O counters

## Summary

Remove the `TypeperfCollector` (which spawns a `typeperf.exe` process per scenario to write `counters.csv`) and replace it with lightweight before/after snapshots of system-wide disk I/O using .NET `PerformanceCounter`. This surfaces two new first-class fields in `run.json` ŌĆö `system_disk_read_bytes` and `system_disk_write_bytes` ŌĆö that capture total machine-wide disk activity during a scenario, including AV-attributed I/O that happens outside the Job Object.

Also removes the Job Object I/O fields (`io_read_bytes`, `io_write_bytes`, `io_read_ops`, `io_write_ops`, `total_processes`) from `run.json` and downstream outputs, as they only measure the workload's own I/O and provide negligible signal for AV impact analysis.

## Motivation

### Why remove typeperf

1. **Process overhead.** Every scenario spawns a `typeperf.exe` child process. On microbench scenarios that run in-process for sub-second durations, the typeperf process startup/shutdown cost is disproportionate.
2. **Rate counters require post-processing.** The CSV contains per-second rate samples. To get totals, someone must parse and sum the CSV ŌĆö no tooling in avbench does this today. The data goes unused.
3. **Not consumed by compare.** `avbench-compare` never reads `counters.csv`. It's purely a manual diagnostic artifact.
4. **Noise source.** The typeperf process itself consumes CPU and I/O, slightly perturbing the measurement it's supposed to diagnose.

### Why remove Job Object I/O fields

The Job Object I/O counters (`io_read_bytes`, `io_write_bytes`, `io_read_ops`, `io_write_ops`) measure only the I/O issued **by processes inside the Job Object** (the build tool and its children). AV overhead I/O ŌĆö signature database reads, scan cache writes, cloud verdict lookups ŌĆö happens in AV service processes (e.g., `MsMpEng.exe`, `ekrn.exe`) which are **outside** the Job Object. These counters are nearly identical between baseline and AV runs, providing no useful signal for AV impact.

`total_processes` is similarly uninformative ŌĆö the build spawns the same number of processes regardless of AV presence.

### Why add system-wide disk counters

The delta of total system disk I/O between baseline and AV runs directly reveals the AV's disk footprint. If a Roslyn clean build generates 3 GB of system-wide disk writes on baseline and 4.5 GB with AV enabled, the 1.5 GB difference is clearly AV-attributed I/O. This is the metric that actually answers "how much extra disk activity did AV cause?"

## Design

### New: `DiskIoSnapshot`

A simple collector that reads cumulative disk byte counters before and after a scenario:

```csharp
// New file: AvBench.Core/Collectors/DiskIoSnapshot.cs

using System.Diagnostics;

namespace AvBench.Core.Collectors;

public sealed class DiskIoSnapshot
{
    private readonly PerformanceCounter _readCounter;
    private readonly PerformanceCounter _writeCounter;
    private long _startRead;
    private long _startWrite;

    public DiskIoSnapshot()
    {
        _readCounter = new PerformanceCounter("PhysicalDisk", "Disk Read Bytes/sec", "_Total", readOnly: true);
        _writeCounter = new PerformanceCounter("PhysicalDisk", "Disk Write Bytes/sec", "_Total", readOnly: true);
    }

    /// <summary>
    /// Capture the cumulative raw counter values before the scenario starts.
    /// </summary>
    public void SnapshotBefore()
    {
        _startRead = _readCounter.NextSample().RawValue;
        _startWrite = _writeCounter.NextSample().RawValue;
    }

    /// <summary>
    /// Capture the cumulative raw counter values after the scenario ends.
    /// Returns the delta (bytes transferred during the scenario).
    /// </summary>
    public (long ReadBytes, long WriteBytes) SnapshotAfter()
    {
        var endRead = _readCounter.NextSample().RawValue;
        var endWrite = _writeCounter.NextSample().RawValue;
        return (endRead - _startRead, endWrite - _startWrite);
    }
}
```

**Key detail:** `PerformanceCounter.NextSample().RawValue` returns the **cumulative** counter value from the kernel, not the per-second cooked rate. The delta between two raw samples gives total bytes transferred during the interval. The counter name says "Bytes/sec" but that refers to the cooked (formatted) display ŌĆö the raw data is cumulative.

### Changes to `ScenarioRunner.RunOnceAsync`

Replace the `TypeperfCollector` block with `DiskIoSnapshot`:

```
Before:
  TypeperfCollector collector = new();
  collector.Start(outputDirectory);
  ... run scenario ...
  collector.Stop();
  collector.Dispose();

After:
  DiskIoSnapshot diskIo = new();
  diskIo.SnapshotBefore();
  ... run scenario ...
  var (readBytes, writeBytes) = diskIo.SnapshotAfter();
  // Store readBytes/writeBytes in execution result
```

No more child process, no more `counters.csv` file.

`IdleChecker` currently uses `typeperf` to sample CPU. Replace with `PerformanceCounter`:

```csharp

for (int i = 0; i < SampleCount; i++)

    await Task.Delay(1000, cancellationToken);
    samples.Add(cpuCounter.NextValue());
}
```

This eliminates the last `typeperf` dependency.

## Risks

| Risk | Mitigation |
|---|---|
| `PerformanceCounter` first-call initialization takes ~50ŌĆō200 ms | One-time cost per scenario. Negligible compared to scenario runtime. Call `SnapshotBefore()` before the timed section. |
| System-wide counters include background OS I/O (indexer, updater) | `IdleChecker` gates on CPU < 20% before the run. Across 3ŌĆō5 sessions, background noise averages out. Absolute values are less important than the baseline-vs-AV delta. |
| `PerformanceCounter("PhysicalDisk", ...)` may not exist on some server SKUs | Windows 10/11 client always has this. Server Core may not have `diskperf` enabled by default. Log a warning and set fields to 0 if the counter is unavailable. |
| Raw counter values could wrap on very long runs | The underlying PERF_COUNTER_LARGE_RAWCOUNT is 64-bit. Wrap is not a practical concern. |
| Removing `counters.csv` loses the per-second diagnostic timeline | The primary use case (total system I/O) is better served by the new first-class fields. If per-second diagnostics are needed in the future, a new milestone can re-add time-series sampling using `PerformanceCounter` in a background task. |
