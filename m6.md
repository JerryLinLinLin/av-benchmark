# Milestone 6 — Fix MOTW benchmark hash-caching weakness & rename scenarios

## Problem

The two MOTW scenarios (`motw-exe-no-motw`, `motw-exe-motw-zone3`) currently copy the **same unmodified `noop.exe`** on every iteration. The executable bytes are identical across all 500 operations.

Many AV products cache scan verdicts by file hash. After the first copy-and-execute, the AV has already hashed the file, classified it, and stored the result. Iterations 2–499 hit that cache and skip deep inspection. The measured latency distribution therefore reflects **one real scan + 499 cache hits**, which underestimates the true per-file MOTW inspection cost.

This is the same problem that `file-write-content` already solves: it patches bytes 0x40–0x43 of the PE template on each iteration so every written file has a unique hash. The MOTW scenarios need the same treatment.

## Comparison with `process-create-wait`

`process-create-wait` also runs the same `noop.exe` 500 times, but that scenario deliberately measures the **cached** process-launch path — a single known binary launched repeatedly from the same location. That is a valid design: it isolates process-creation overhead on a warm cache.

The MOTW scenarios have a different purpose. They measure whether the presence of `Zone.Identifier` changes the cost of executing a newly arrived file. "Newly arrived" implies the AV has not seen the file before, which means the hash should be unique each time. The current implementation contradicts the scenario's own premise.

The scenario names are also misleading. `motw-exe-no-motw` defines itself as the negation of something it doesn't do instead of describing its actual purpose: executing a never-before-seen binary. `motw-exe-motw-zone3` buries the key distinction in a long compound name.

## Proposed changes

### Rename scenario IDs

| Old ID | New ID | Rationale |
|---|---|---|
| `motw-exe-no-motw` | `new-exe-run` | Describes what it actually tests: run a new unknown exe |
| `motw-exe-motw-zone3` | `new-exe-run-motw` | Same base name + MOTW suffix makes the A/B relationship obvious |

This rename touches the factory, the worker routing switch, and the scenario ID strings.

### `MicrobenchWorker.cs` — `ExecuteMotw`

Patch the copied executable's bytes before (optionally) stamping MOTW and before execution, so the AV sees a unique hash on every iteration.

**Current loop body** (simplified):

```csharp
for (var index = 0; index < totalOperations; index++)
{
    var opDirectory = Path.Combine(root, $"bench_{index:D5}");
    Directory.CreateDirectory(opDirectory);

    foreach (var supportFile in supportFiles)
        File.Copy(supportFile, Path.Combine(opDirectory, Path.GetFileName(supportFile)), overwrite: true);

    if (applyMotw)
        File.WriteAllText(destExe + ":Zone.Identifier", "[ZoneTransfer]\r\nZoneId=3\r\n");

    RunExecutable(destExe);
    FileSystemUtil.DeletePathIfExists(opDirectory);
    histogram.Record(Stopwatch.GetTimestamp() - start);
}
```

**Proposed loop body:**

```csharp
for (var index = 0; index < totalOperations; index++)
{
    var opDirectory = Path.Combine(root, $"bench_{index:D5}");
    Directory.CreateDirectory(opDirectory);

    foreach (var supportFile in supportFiles)
        File.Copy(supportFile, Path.Combine(opDirectory, Path.GetFileName(supportFile)), overwrite: true);

    // Patch 4 bytes so every copy has a unique hash, defeating AV scan caching.
    PatchIterationBytes(destExe, index);

    if (applyMotw)
        File.WriteAllText(destExe + ":Zone.Identifier", "[ZoneTransfer]\r\nZoneId=3\r\n");

    RunExecutable(destExe);
    FileSystemUtil.DeletePathIfExists(opDirectory);
    histogram.Record(Stopwatch.GetTimestamp() - start);
}
```

### New shared helper: `PatchIterationBytes`

A small static method that opens the copied exe, writes the iteration index into bytes 0x40–0x43 (the same PE header padding region `file-write-content` already uses), and closes the file. This is a one-line semantic change — the helper can live in `MicrobenchWorker.cs` alongside the existing methods.

```csharp
private static void PatchIterationBytes(string exePath, int index)
{
    using var stream = new FileStream(exePath, FileMode.Open, FileAccess.Write);
    stream.Seek(0x40, SeekOrigin.Begin);
    Span<byte> buf = stackalloc byte[4];
    BinaryPrimitives.WriteInt32LittleEndian(buf, index);
    stream.Write(buf);
}
```

### Why bytes 0x40–0x43

Offset 0x40 in a PE file is inside the DOS stub's reserved area (between the DOS header and the PE signature pointer at 0x3C). Overwriting these bytes changes the file hash without corrupting the PE structure — the same approach `file-write-content` already uses and validates on every run.

## Impact on results

After this change, the MOTW scenarios will measure the **uncached** inspection path on every iteration. Expected effects:

- **Higher baseline latency** for both scenarios (AV must hash/scan every copy)
- **Larger delta** between `new-exe-run` and `new-exe-run-motw` if the AV applies extra checks for MOTW-marked files specifically
- **More realistic tail latency** (p95/p99) that reflects actual first-encounter costs rather than cache-hit fast paths
- **Better parity** with `file-write-content`, which already defeats caching the same way

Previous results from before this change are **not comparable** to results after it. The scenario IDs have changed (`motw-exe-no-motw` → `new-exe-run`, `motw-exe-motw-zone3` → `new-exe-run-motw`), so old and new data will not accidentally merge in the comparison tool.
