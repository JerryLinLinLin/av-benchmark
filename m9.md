# Milestone 9 — Baseline CV, disk I/O in summary, median-based slowdown, and cleaner summary table

## Problem 1: Baseline CV is not computed or displayed

`CompareEngine` computes `cv_pct` from the AV side's wall-time samples only. The baseline's CV is never calculated. If the baseline itself is noisy for a scenario, the `slowdown_pct` is unreliable even when the AV side looks clean — but the report gives no indication of this.

### Proposed change

Compute `baseline_cv_pct` from the baseline wall-time samples, the same way `cv_pct` is computed for the AV side. Add it to `ComparisonRow`, `compare.csv`, and `summary.md`.

The `status` field should incorporate baseline noise: if either `cv_pct` or `baseline_cv_pct` exceeds the noisy threshold (10%), mark the row as `noisy`.

### Files to modify

| File | Change |
|---|---|
| `CompareEngine.cs` | Compute `baseline_cv_pct` from baseline wall samples; factor it into `status` |
| `ComparisonRow` (in `CompareEngine.cs`) | Add `BaselineCvPct` property |
| `CompareCsvWriter.cs` | Add `baseline_cv_pct` column |
| `SummaryRenderer.cs` | Add `Baseline CV %` column to the table |

## Problem 2: Disk read/write deltas are not in the summary table

`summary.md` only mentions disk I/O in the footnote callouts ("Largest system disk write delta: ..."). The per-scenario table has no disk columns, so you can't see disk overhead for every scenario at a glance. The data exists in `compare.csv` but not in the human-readable report.

### Proposed change

Add two columns to the `summary.md` table:

| New column | Value |
|---|---|
| `Disk Read Δ (MB)` | `system_disk_read_mb - baseline_system_disk_read_mb` |
| `Disk Write Δ (MB)` | `system_disk_write_mb - baseline_system_disk_write_mb` |

These show the delta, not the absolute values — the delta is what matters for AV analysis. Absolute values remain in `compare.csv` for anyone who needs them.

### Files to modify

| File | Change |
|---|---|
| `SummaryRenderer.cs` | Add two disk delta columns to the Markdown table |

## Problem 3: Slowdown should use median, not mean

`slowdown_pct` is always computed from `mean_wall_ms`. When CV is high, the mean is pulled by outliers — the median is more robust. A threshold-based approach (use mean when clean, median when noisy) introduces a discontinuity at the threshold boundary and can mix different statistics across the AV and baseline sides.

### Proposed change

Always compute `slowdown_pct` from `median_wall_ms` on both sides:

```
slowdown_pct = (median_wall_ms - baseline_median_wall_ms) / baseline_median_wall_ms * 100
```

Why always-median is better than a threshold switch:

- **No discontinuity**: no magic threshold where the metric jumps between two different statistics.
- **No mixed comparisons**: same statistic on both sides, always.
- **When data is clean** (low CV), median ≈ mean — you lose nothing.
- **When data is noisy**, median is strictly more robust than mean.
- **Simpler**: no `representative_wall_ms` field, no conditional logic.
- With 3 sessions, median is the middle value — the most outlier-resistant single number at that sample size.

`compare.csv` continues to emit both `mean_wall_ms` and `median_wall_ms` so downstream tools can use whichever they prefer. `summary.md` shows `Median Wall (ms)` as the primary wall-time column.

### Files to modify

| File | Change |
|---|---|
| `CompareEngine.cs` | Compute `SlowdownPct` from `MedianWallMs` and baseline median instead of means |
| `SummaryRenderer.cs` | Show `MedianWallMs` in the wall column, label it `Median Wall (ms)` |

## Problem 4: Kernel CPU columns are noise in the summary table

The summary table currently has 3 kernel CPU columns (`Kernel CPU %`, `Baseline Kernel %`, `Kernel Shift`). These are only populated for the 4 compilation scenarios — all 27 microbenchmarks show `0.0% | 0.0% | +0.0pp` because they run in-process without Job Object accounting.

Even for compilation scenarios, the kernel CPU shift is a weak signal. It's a ratio (not absolute), it only reflects CPU inside the workload's process tree (missing AV service processes), and the shifts are small (+0.5 to +1.4pp in exp1).

### Proposed change

Remove the 3 kernel CPU columns from the `summary.md` table. Instead, report kernel CPU shift as a **footnote callout** (same pattern as disk I/O callouts), only when the shift exceeds a meaningful threshold. The data stays in `compare.csv` for detailed analysis.

This also applies to `peak_memory_mb` — it is zero for all microbenchmarks and rarely affected by AV. Keep it in `compare.csv` only, not in the summary table.

### Files to modify

| File | Change |
|---|---|
| `SummaryRenderer.cs` | Remove 3 kernel CPU columns from table; kernel shift becomes a footnote callout (already exists, just keep it) |

## Problem 5: Microbench percentile comparison is missing

`run.json` contains per-microbench latency percentiles (`p50_us`, `p95_us`, `p99_us`, `max_us`), but `CompareEngine` ignores them entirely. It only compares `wall_ms`. For microbenchmarks, tail latency slowdown is often more revealing than total wall time — a scenario can show +20% wall slowdown but +200% p99 slowdown, meaning AV adds rare but severe stalls. That's the more actionable finding.

### Proposed change

For microbench scenarios, compute median p50/p95/p99 across sessions (same median-based approach as wall time), then derive percentile slowdown versus baseline:

```
p95_slowdown_pct = (median_p95_us - baseline_median_p95_us) / baseline_median_p95_us * 100
```

Add to `ComparisonRow`:

| Property | Type |
|---|---|
| `MedianP50Us` | `double` |
| `MedianP95Us` | `double` |
| `MedianP99Us` | `double` |
| `BaselineMedianP95Us` | `double` |
| `P95SlowdownPct` | `double` |

Add `p95_slowdown_pct` to the `summary.md` table for microbench rows (shows `—` for compilation scenarios). Add all percentile columns to `compare.csv`.

### Files to modify

| File | Change |
|---|---|
| `CompareEngine.cs` | Read `p50_us`, `p95_us`, `p99_us` from microbench results; compute median across sessions; derive `P95SlowdownPct` |
| `ComparisonRow` | Add percentile properties |
| `CompareCsvWriter.cs` | Add percentile columns |
| `SummaryRenderer.cs` | Add `p95 Slowdown` column |

## Problem 6: No cross-AV comparison table

When comparing 2+ AV products, the summary renders separate sections per AV. To answer "which AV is worst for this scenario?", you have to mentally match rows across sections.

### Proposed change

When there are 2+ AV configurations in the input, append a **cross-AV comparison table** at the end of `summary.md`:

```
## Cross-AV comparison

| Scenario          | baseline (ms) | defender     | eset         |
|-------------------|---------------|--------------|--------------|
| dll-load-unique   | 8,001         | +597.1%      | +299.9%      |
| new-exe-run-motw  | 1,931         | +3566.3%     | +2230.5%     |
| file-create-delete| 1,494         | +50.3%       | +20.4%       |
| ...               |               |              |              |
```

The baseline column shows the median wall time. Each AV column shows `slowdown_pct`. Rows are sorted by the worst slowdown across any AV. Noisy/failed/insufficient entries are marked with `*` and a legend at the bottom.

This table is only appended when `rows.GroupBy(row => row.AvName)` produces 2+ groups. With a single AV, the existing per-AV section is sufficient.

### Files to modify

| File | Change |
|---|---|
| `SummaryRenderer.cs` | After per-AV sections, append cross-AV table when 2+ AV groups exist |

## Problem 7: Negative slowdowns are not flagged as anomalies

exp1 shows `wmi-query` at -35.8% and `file-copy-large` at -9.0% — AV apparently making things *faster*. These are suspicious (caching artifacts, measurement noise, or system state differences). Currently they're presented without any warning.

### Proposed change

Add an `anomaly` status. A row is flagged as `anomaly` when:

- `status` would otherwise be `ok` (not failed, not noisy)
- `slowdown_pct < -10%`

The status priority becomes: `failed` > `noisy` > `anomaly` > `ok`.

In the summary, anomaly rows are called out in a footnote: "Anomaly scenarios (AV appears faster — likely caching artifact): wmi-query, ..."

### Files to modify

| File | Change |
|---|---|
| `CompareEngine.cs` | Add `anomaly` status when slowdown is significantly negative and data is otherwise clean |
| `SummaryRenderer.cs` | Add anomaly footnote callout |

## Problem 8: Single-session results show misleading `ok` status

With 1 session, `cv_pct` is computed as 0.0 (the `wallSamples.Length > 1` check returns 0). Status shows `ok`. That's misleading — a single session provides no variance information at all. The `ok` status implies the data was verified as stable when it was never tested.

### Proposed change

Add a session-count check: if `sessions < 3` (for either AV or baseline), mark the row as `insufficient` regardless of other checks.

Status priority becomes: `failed` > `insufficient` > `noisy` > `anomaly` > `ok`.

In the summary, insufficient rows are called out: "Insufficient data (< 3 sessions): ..."

### Files to modify

| File | Change |
|---|---|
| `CompareEngine.cs` | Add `insufficient` status when session count < 3 on either side |
| `SummaryRenderer.cs` | Add insufficient footnote callout |
