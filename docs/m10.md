# M10 – Auto-exclude single outlier from noisy scenarios

## Problem

Some scenarios get flagged `noisy` (CV > 10%) because of a single anomalous
run.  For example `new-exe-run-motw` under ms-defender had wall_ms values of
89210, 54980, 68200, 56433, 63715 — one outlier inflates CV to 18.5%.  Dropping
that single run brings CV to ~8.7% and the remaining data is usable.

Today the compare engine treats all successful runs equally, so the entire
scenario is discarded as noisy even when 4 out of 5 runs agree.

## Proposed behaviour

In `CompareEngine`, after collecting `successfulRuns` for a scenario:

1. Compute CV from wall-time samples as today.
2. If CV ≤ 10% → keep all runs, proceed as today.
3. If CV > 10% and there are ≥ 4 successful runs → try excluding the single
   furthest-from-median run (largest |wall − median|).  Re-compute CV on the
   remaining N−1 runs.
   - If the reduced CV ≤ 10% → use the N−1 runs for **all** aggregation of
     that scenario (wall, cpu, kernel, disk, p50/p95/p99).  Record
     `ExcludedRuns = 1`.
   - If the reduced CV > 10% → keep all N runs, mark status `noisy` as today.
     Do **not** attempt to remove a second run.
4. If CV > 10% and there are < 4 successful runs → mark `noisy` (not enough
   data to safely drop one).

The same logic applies independently to **baseline** runs and to **AV** runs.
Each side is filtered on its own.

## ComparisonRow changes

Add two fields:

```csharp
public int ExcludedRuns { get; init; }        // 0 or 1
public int BaselineExcludedRuns { get; init; } // 0 or 1
```

`Sessions` / `BaselineSessions` keep the **total** count (including the
excluded run) so the reader knows how many were attempted.

## Implementation sketch

Add a private helper in `CompareEngine`:

```csharp
private static List<T> TryExcludeOutlier<T>(
    List<T> runs,
    Func<T, double> wallSelector,
    double noisyThreshold,
    out int excluded)
{
    excluded = 0;
    var samples = runs.Select(wallSelector).ToArray();
    if (samples.Length < 4)
        return runs;

    var cv = StandardDeviation(samples) / samples.Average() * 100.0;
    if (cv <= noisyThreshold)
        return runs;

    // Find run furthest from median
    var median = Median(samples);
    var worstIndex = 0;
    var worstDistance = 0.0;
    for (var i = 0; i < samples.Length; i++)
    {
        var distance = Math.Abs(samples[i] - median);
        if (distance > worstDistance)
        {
            worstDistance = distance;
            worstIndex = i;
        }
    }

    // Re-check CV without the outlier
    var remaining = runs.Where((_, i) => i != worstIndex).ToList();
    var remainingSamples = remaining.Select(wallSelector).ToArray();
    var remainingCv = StandardDeviation(remainingSamples)
                    / remainingSamples.Average() * 100.0;

    if (remainingCv <= noisyThreshold)
    {
        excluded = 1;
        return remaining;
    }

    return runs;  // still noisy, keep original set
}
```

Call site (AV side, around line 56):

```csharp
successfulRuns = TryExcludeOutlier(
    successfulRuns,
    static run => (double)run.WallMs,
    NoisyThresholdPct,
    out var excludedRuns);
```

Same pattern for `baselineScenarioRuns` → `out var baselineExcludedRuns`.

The rest of the aggregation code stays untouched — it already reads from
`successfulRuns` / `baselineScenarioRuns`.

## CSV output

`CompareCsvWriter` adds two columns: `excluded_runs`, `baseline_excluded_runs`.

## Summary rendering

`SummaryRenderer`:
- When `ExcludedRuns > 0`, append a footnote marker (†) to the scenario row.
- Add a footnote line at the bottom: `† 1 outlier run excluded to reduce CV`.
- If the scenario is *still* noisy after the attempt, no footnote — it just
  stays `noisy` as before.

## What this does NOT do

- Never excludes more than 1 run per side per scenario.
- Never excludes if < 4 successful runs remain after exclusion.
- Does not touch `runs.csv` or raw data — exclusion is only in the compare
  aggregation layer.
