# M11 – Fix pipe-roundtrip noise & add incremental recompare

Two independent changes.

## 1. Stabilise `pipe-roundtrip`

### Problem

`pipe-roundtrip` has a baseline CV of ~24%, flagged `noisy` in every
comparison.  The root cause is the current implementation: it creates a new
named pipe **and** a new `Task.Run` server thread on every iteration.  Each
measurement therefore includes pipe creation, connection handshake, and
thread-pool scheduling — not just the data round-trip.  The thread-pool
scheduling is inherently jittery and dominates variance.

Evidence from the 5 baseline runs:

| Run | wall_ms | p50 (µs) | p95 (µs) | mean latency (µs) |
|-----|---------|----------|----------|--------------------|
| 1   | 2096    | 122      | 13 243   | 1025               |
| 2   | 1519    | 137      | 407      | 733                |
| 3   | 1036    | 115      | 229      | 505                |
| 4   | 1741    | 132      | 978      | 839                |
| 5   | 1253    | 111      | 311      | 608                |

p50 is stable (111–137 µs).  p95 swings from 229 µs to 13 243 µs.
The p99 is always ~15–16 ms regardless of run, confirming a persistent heavy
tail from OS thread scheduling.  A few extra operations hitting that tail in
one run versus another shift total wall time by hundreds of milliseconds on a
~1.5 s workload.

### Proposed changes

Three changes to `ExecutePipeRoundtrip` in `MicrobenchWorker.System.cs`:

**a) Reuse a single pipe for the entire loop.**  Create one
`NamedPipeServerStream` / `NamedPipeClientStream` pair, connect once, then
loop N round-trips over the established pipe.  This removes per-iteration
creation/connection overhead and makes the benchmark measure what its name
says: pipe data round-trip latency.

**b) Use a dedicated `Thread` instead of `Task.Run`.**  The server side should
run on a plain `Thread { IsBackground = true }` rather than a thread-pool
work item.  This eliminates thread-pool scheduling variance from the
measurement.

**c) Add a warmup phase.**  Run 100 unmeasured round-trips before the
recorded loop to let the JIT, OS pipe buffers, and thread scheduler settle.

### Implementation sketch

```csharp
private static MicrobenchMetrics ExecutePipeRoundtrip(int totalOperations)
{
    var payload = new byte[4096];
    var response = new byte[payload.Length];
    Random.Shared.NextBytes(payload);

    const int warmup = 100;
    var pipeName = $"avbench-pipe-{System.Environment.ProcessId}";

    using var server = new NamedPipeServerStream(
        pipeName, PipeDirection.InOut, 1,
        PipeTransmissionMode.Byte, PipeOptions.None);

    using var client = new NamedPipeClientStream(
        ".", pipeName, PipeDirection.InOut, PipeOptions.None);

    var serverThread = new Thread(() =>
    {
        var buffer = new byte[payload.Length];
        for (var i = 0; i < warmup + totalOperations; i++)
        {
            if (i == 0) server.WaitForConnection();
            ReadExact(server, buffer, buffer.Length);
            server.Write(buffer, 0, buffer.Length);
            server.Flush();
        }
    }) { IsBackground = true };
    serverThread.Start();

    client.Connect(5_000);

    // Warmup (unmeasured)
    for (var i = 0; i < warmup; i++)
    {
        client.Write(payload, 0, payload.Length);
        client.Flush();
        ReadExact(client, response, response.Length);
    }

    // Measured loop
    var histogram = new LatencyHistogram(totalOperations);
    var stopwatch = Stopwatch.StartNew();

    for (var index = 0; index < totalOperations; index++)
    {
        var start = Stopwatch.GetTimestamp();
        client.Write(payload, 0, payload.Length);
        client.Flush();
        ReadExact(client, response, response.Length);
        histogram.Record(Stopwatch.GetTimestamp() - start);
    }

    stopwatch.Stop();
    serverThread.Join(5_000);
    return BuildMetrics(1, totalOperations, stopwatch.Elapsed, histogram);
}
```

### Operation count

Keep at 2 000.  With creation overhead removed each iteration is much faster,
but the total is still enough for stable percentile data and the suite
runtime increase is negligible.

### Documentation

Update the `pipe-roundtrip` row in `docs/workloads.md`:

> Creates a named-pipe server/client pair, then exchanges 4 KB round-trips
> 2 000 times over the established connection.  Measures steady-state local
> IPC latency.

### What this does NOT do

- Does not add a new `pipe-create-teardown` scenario.  If that turns out to
  be useful it can be added later as a separate scenario.
- Does not change the operation count or payload size.
- Does not touch any other scenario.

## 2. Incremental recompare: `avbench-compare --rebuild`

### Problem

Today, replacing a single scenario `run.json` (for example after re-running
just `pipe-roundtrip`) requires manually reconstructing the full comparison.
The compare tool reads `run.json` files directly and does not update the
per-session `runs.csv`.  There is no way to:

1. Regenerate `runs.csv` from the `run.json` files that already exist in a
   results directory.
2. Then recompute `compare.csv` and `summary.md` from those updated sessions.

That means any single-scenario re-run forces either a full suite re-run or
careful manual file surgery.

### Proposed behaviour

Add a `--rebuild` flag to `avbench-compare`.  When set, the tool performs a
three-step chain before the normal comparison:

```
Step 1:  For each results directory (baseline + all inputs),
         scan every <scenario>/run.json and regenerate runs.csv
         from those files.

Step 2:  Run the normal comparison engine on the run.json data
         (same as today).

Step 3:  Write compare.csv and summary.md to --output
         (same as today).
```

Without `--rebuild`, behaviour is identical to today — `runs.csv` is not
touched.

### Workflow it enables

```
# Original full run already completed.  Results live in:
#   data/exp1/baseline-os/results3/
#   data/exp1/huorong/results3/

# Re-run only pipe-roundtrip in the baseline VM:
avbench run --workload pipe-roundtrip --output C:\results3

# Copy the new run.json back:
copy C:\results3\pipe-roundtrip\run.json ^
     data\exp1\baseline-os\results3\pipe-roundtrip\run.json

# Rebuild everything from the updated run.json files:
avbench-compare ^
    --baseline data\exp1\baseline-os ^
    --input data\exp1\huorong data\exp1\ms-defender ^
    --output data\exp1 ^
    --rebuild
```

The `--rebuild` flag regenerates every `runs.csv` it finds, then produces
fresh `compare.csv` and `summary.md` — all from whatever `run.json` files
are on disk.

### Code changes

**a) `CompareCommand.cs` — add `--rebuild` option.**

```csharp
var rebuildOption = new Option<bool>("--rebuild")
{
    Description = "Regenerate runs.csv in each results directory from run.json "
                + "files before comparing."
};
command.Options.Add(rebuildOption);
```

Read the flag in the handler:

```csharp
var rebuild = parseResult.GetValue(rebuildOption);
```

**b) `CompareCommand.cs` — rebuild logic before comparison.**

When `--rebuild` is set, after loading runs from each directory, also
regenerate `runs.csv` in every results subdirectory that contains at least
one `run.json`.

```csharp
if (rebuild)
{
    RebuildRunsCsv(baselineDirectory.FullName);
    foreach (var inputDirectory in inputDirectories)
    {
        RebuildRunsCsv(inputDirectory.FullName);
    }
}
```

The helper finds each `resultsN/` subdirectory (directories that contain
`run.json` files), collects all `run.json` files in that subdirectory,
deserialises them, and rewrites `runs.csv` using the existing
`CsvResultWriter`:

```csharp
private static void RebuildRunsCsv(string rootDirectory)
{
    // Group run.json files by their parent results directory.
    // Structure: rootDirectory/results1/<scenario>/run.json
    //            rootDirectory/results2/<scenario>/run.json
    var runFiles = Directory.EnumerateFiles(rootDirectory, "run.json", SearchOption.AllDirectories);
    var bySession = new Dictionary<string, List<RunResult>>(StringComparer.OrdinalIgnoreCase);

    foreach (var path in runFiles)
    {
        // The session directory is two levels up: <scenario>/run.json -> resultsN/
        var scenarioDir = Path.GetDirectoryName(path)!;
        var sessionDir = Path.GetDirectoryName(scenarioDir)!;

        var json = File.ReadAllText(path);
        var run = JsonSerializer.Deserialize(json, AvBenchJsonContext.Default.RunResult);
        if (run is null) continue;

        if (!bySession.TryGetValue(sessionDir, out var list))
        {
            list = [];
            bySession[sessionDir] = list;
        }
        list.Add(run);
    }

    foreach (var (sessionDir, runs) in bySession)
    {
        var csvPath = Path.Combine(sessionDir, "runs.csv");
        CsvResultWriter.WriteAsync(runs, csvPath, CancellationToken.None)
            .GetAwaiter().GetResult();
        Console.WriteLine($"[rebuild] Wrote {runs.Count} rows to {csvPath}");
    }
}
```

**c) No changes to `CompareEngine`, `CompareCsvWriter`, or `SummaryRenderer`.**

The comparison pipeline already reads from `run.json` directly.  The
`--rebuild` step only brings `runs.csv` back into sync so the per-session
CSV stays consistent with the JSON files.  The comparison output path is
unchanged.

### What this does NOT do

- Does not make `runs.csv` the source of truth.  `run.json` remains the
  primary data format; `runs.csv` is a derived convenience file.
- Does not add incremental / single-scenario diffing.  The compare engine
  still processes all scenarios in all sessions every time.
- Does not add a `--scenario` filter to compare.  That could be useful but
  is a separate change.
- Does not modify `avbench run`.  The runner still writes both `run.json`
  and `runs.csv` as today; `--rebuild` is only needed after manual
  `run.json` replacement.
