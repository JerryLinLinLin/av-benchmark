# Milestone 8 — Reduce new-exe-run operation count & fix scenario order

## Problem 1: `new-exe-run` / `new-exe-run-motw` at 500 operations is too many

After M6, every iteration produces a unique file hash (4-byte patch). The AV must do a full scan on each copy — no cache hits. Some AV products perform cloud reputation lookups and extended analysis on unknown executables, which can take 500 ms–2 s per iteration. At 500 iterations that is 4–17 minutes per scenario, times 2 scenarios = 8–34 minutes just for this pair.

Compare with `process-create-wait`, which also runs 500 times but uses the **same cached binary** every time — the AV scans once and caches, so per-iteration cost is low. The two scenario pairs have fundamentally different per-iteration cost profiles now and should have different operation counts.

### Proposed change

Reduce `MotwOperations` from `500` to `50`.

50 iterations is still enough for meaningful latency percentiles on a heavy per-operation path (process create + full AV scan + potential cloud lookup). This is the same tier as `archive-extract` (10 iterations) and `file-copy-large` (10 iterations) — scenarios where each operation is individually expensive.

### Files to modify

| File | Change |
|---|---|
| `AvBench.Core/Scenarios/MicrobenchScenarioFactory.cs` | `MotwOperations = 500` → `MotwOperations = 50` |

One line.

## Problem 2: Scenario run order should be fully deterministic

Currently `OrderScenariosForSession` applies a Fisher-Yates shuffle within the microbench phase and within the compilation phase. The original rationale was to distribute systematic ordering effects (AV cache warm-up, thermal drift) across sessions.

In practice this causes more problems than it solves:

- **Reproducibility**: Two sessions from the same VM snapshot produce results in different orders, making it harder to compare logs side-by-side.
- **Cooldowns already mitigate bleed-over**: M6 added 10 s / 20 s cooldown pauses between scenarios, which address the AV background activity concern that the shuffle was designed for.
- **Ordering effects are consistent, not eliminated**: Shuffling doesn't remove ordering effects — it randomizes them. A fixed order makes ordering effects consistent across sessions, which means they cancel out in baseline-vs-AV comparisons (both sessions see the same order).

### Proposed change

Remove the Fisher-Yates shuffle entirely. Run scenarios in the fixed order defined by `AllScenarioIds` in `MicrobenchScenarioFactory`, followed by compilation scenarios in their factory-defined order (ripgrep then roslyn).
