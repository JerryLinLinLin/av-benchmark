# 两份编译跟踪，拆穿了 AV 引擎的什么底

你有没有发现，同一台机器上，杀软能把某个项目的编译拖得很慢，另一个却几乎没感觉？答案通常不是"项目越大越慢"这么简单。真正起作用的是构建系统的类型、编译器工具链的形态、以及 I/O 流量的结构——而这些差异恰好会戳中不同 AV 产品的不同软肋。

我们抓了两份 Process Monitor 跟踪——一份是编译 [ripgrep](https://github.com/BurntSushi/ripgrep)（Rust 命令行工具），另一份是编译 [Roslyn](https://github.com/dotnet/roslyn)（微软的 C#/VB 编译器平台）——然后站在 AV 的视角去拆解它们。我们关心的不是"哪个编得慢"，而是"一个挂在文件系统上实时做信任判断的安全产品，看到这两个构建时分别在忙什么？"

分析基于核心构建进程树：以基准测试工具 (`avbench.exe`) 为根，沿子进程向下追踪，借此过滤掉 ProcMon 自身噪声、桌面无关活动和系统服务。详细表格见 `analysis/compilation-procmon-analysis.md`，流水线说明见 `analysis/workload-profile-pipeline.md`，机器可读数据见 `analysis/procmon-summary.json`。

## 先说结论

Roslyn 不是"放大版的 ripgrep"。在操作系统这一层看，它俩根本不是同一类工作负载。

**Ripgrep** 小而精，走的是原生工具链路线——大约 20.2 万个核心构建事件，涉及约 3,400 条唯一文件路径。活动高度集中在 `cargo.exe`、`rustc.exe`、`link.exe`、构建脚本、PDB、导入/静态库和刚产出的原生文件上。可以理解为一场密集的构建产物生成考试：路径不多，但每一条都直戳 AV 的痛点。

**Roslyn** 则是一张大规模托管构建图——约 1,260 万个核心事件，涉及约 15.38 万条唯一路径。跟踪会铺开到项目文件、`.props`/`.targets`、源文件、引用程序集、NuGet 缓存、SDK 文件、分析器 DLL、生成代码，以及 `VBCSCompiler.exe` 编译器服务器。这是一场路径扇出和元数据压力的大考，再加上一大波新生成 DLL 带来的二次冲击。

虽然都叫"编译"，但它们考的是 AV 产品完全不同的能力：一个考原生产物创建，一个考大规模托管构建图遍历。

## 先说背景

[Ripgrep](https://github.com/BurntSushi/ripgrep) 是一个面向大型代码库和日常开发的递归正则搜索工具。用 [Cargo](https://doc.rust-lang.org/cargo/commands/cargo-build.html) 构建它，意味着编译本地包、依赖包，以及 crate 定义的所有[构建脚本](https://doc.rust-lang.org/cargo/reference/build-scripts.html)。

[Roslyn](https://github.com/dotnet/roslyn) 是开源的 C#/VB 编译器平台。对它跑 [`dotnet build`](https://learn.microsoft.com/en-us/dotnet/core/tools/dotnet-build)，会拉入 [MSBuild](https://learn.microsoft.com/en-us/visualstudio/msbuild/incremental-builds?view=visualstudio)、解决方案和项目求值、[NuGet](https://learn.microsoft.com/en-us/nuget/concepts/msbuild-props-and-targets) 包状态、SDK 导入、增量输入/输出检查，以及编译器服务器行为。所以 Roslyn 的跟踪里才会塞满存在性检查、时间戳探测、打开/查询操作、引用读取和生成输出。

## 真正的发现

说"Roslyn 更大"没错，但这话跟没说一样。真正有意思的发现是：ripgrep 和 Roslyn 戳到的是 *AV 决策链上不同的环节*。

Ripgrep 问的是一个很聚焦的原生构建问题：可信的开发工具在短时间内集中产出一批新的原生产物——PDB、导入库、Rust 元数据、可执行文件——你扛不扛得住？

Roslyn 问的则是一个广度极大的托管构建问题：构建系统需要在庞大的项目文件、MSBuild 导入、SDK 文件、引用程序集、NuGet 资产、分析器 DLL、生成代码和新 DLL 输出的图谱中反复打开、探测、读取、写入——你跟不跟得上？

这是两道完全不同的安全考题。一个产品可以漂亮地搞定其中一个，却在另一个上翻车——而且不是意外。比如：一个 Microsoft SDK 允许列表做得好的产品，Roslyn 的大部分输入侧可能一路绿灯，但新生成的本地 DLL 一落地就卡住了。再比如：一个元数据查询过滤很高效的产品，扛得住 Roslyn 的项目图，却因为要认真检查刚生成的原生可执行文件而拖慢 ripgrep。

## 为什么偏偏选这一对

"编译"不是单一工作负载——它是一整个家族。一次真实的开发者构建可能涉及依赖发现、源文件读取、元数据检查、编译器进程树、映像加载、链接器输出、DLL/EXE 写入、包缓存、云端信誉查询，以及围绕开发工具的策略判断。Ripgrep 和 Roslyn 落在这个空间里截然不同的角落。

**Ripgrep 作为原生构建代表：**

- 它是一个真实应用，不是"hello world"式的合成测试。Cargo 编排依赖图并反复调用 `rustc`，工作负载包含真实的包图编排。
- 在 Windows/MSVC 上，最终路径包括原生链接器/资源工具、PDB、导入库、对象文件和可执行文件创建。
- 构建规模足够小，AV 开销不会被庞大构建系统的基线淹没。
- 它给 AV 出的题很直接：你会不会惩罚刚生成的原生输出和链接器/编译器产物？

**Roslyn 作为托管构建代表：**

- 大型真实 .NET 编译器平台仓库，包含大量项目、源文件、资源、生成文件、分析器、测试和引用依赖。
- `dotnet build` 走 MSBuild，天然会带上项目求值、NuGet/缓存行为、SDK 解析和 target/import 处理。
- MSBuild 的增量逻辑靠的是输入/输出关系和时间戳——就算有些活儿能跳过，也得先检查一大堆文件才知道能不能跳。
- 跟踪捕获了 `VBCSCompiler.exe`，因此包含编译器服务器的输入扫描行为，而不只是短命的编译器进程。
- 它给 AV 出的题更宽：你能不能在巨大的文件图扇出和海量托管程序集输出面前，把每次打开/查询的开销控制住？

两个放在一起才有意义。单用 ripgrep，主要考的是原生产物创建和 Rust/MSVC 工具链行为。单用 Roslyn，主要考的是 MSBuild/NuGet/引用程序集遍历和托管 DLL 输出。只有合在一起，才能覆盖两类"在开发者眼里是同一件事、在 AV 眼里*完全不是一回事*"的场景。

## 深入底层：为什么跟踪会分叉

差异从构建系统层面就开始了。

Cargo 干的活儿相对直接：编译包及其依赖。在这份跟踪里，就是 `cargo.exe` 编排、一堆 `rustc.exe` 进程、构建脚本，外加 MSVC 链接器/资源工具。构建的核心动作是编译 crate、然后往 `target\release` 下写产物。只要 Rustup 和 Cargo 缓存到位，涉及的路径空间基本可控。

MSBuild 管的事就多得多了。对一整个解决方案跑 `dotnet build`，得求值项目文件、导入的 `.props` 和 `.targets`、SDK 文件、包资产、目标框架、分析器配置、引用程序集、生成输出，还得判断每组输入/输出是不是过期的。编译还没真正开始，它就已经不停地在问文件系统：这个文件存在吗？哪个 target 生效？哪个引用胜出？这个输出过期了吗？所以哪怕一个 .NET 构建最终产出的二进制只占触达路径的一小部分，`CreateFile`、`QueryOpen`、`QueryDirectory` 和 `QueryNetworkOpenInformationFile` 事件还是会铺天盖地。

Roslyn 之所以放大了这种效应，是因为它*本身就是*一个编译器平台。C# 和 VB 编译器、工作区、分析器、测试、源生成器、资源、`.editorconfig` 和 `.globalconfig` 输入、引用程序集、多目标输出——全堆在一个仓库里。这不是"编译很多 `.cs` 文件"那么简单，而是"求值一个巨大的 .NET 构建图、解析海量引用、跑编译器基础设施、输出一大批托管程序集"。

Ripgrep 则把另一个方向拉到极端。Rust 编译会创建 crate 元数据和库（`.rmeta`、`.rlib`），从 Rustup 工具链库读取，在 Windows/MSVC 上写 PDB 和原生库，最后调用链接器/资源工具生成最终输出。与其说路径扇出大，不如说是高度集中的产物写入。

## 五个关键维度的真正差异

**1. 路径扇出 vs. 产物集中。** Roslyn 触达约 15.38 万条唯一文件路径，ripgrep 约 3,400 条。这 45 倍的差距不只是"量更大"——它从根本上改变了安全问题的性质。每多一条不同路径，AV 引擎就多一个决策点：这个哈希见过吗？签名方可信吗？这是包文件还是新生成的输出？路径集一爆炸，热缓存的假设就越来越靠不住了。

**2. 元数据压力。** Roslyn 的打开/查询操作数以百万计，因为 MSBuild 和 .NET SDK 解析本质上就是"先问一遍文件系统再决定干什么"的路径决策系统。构建系统得把文件系统翻个底朝天，才知道该编译、复制、跳过还是生成什么。Ripgrep 也有元数据操作，但不像 Roslyn 那样被构建图求值主导。

**3. 输出类型。** Ripgrep 创建的类可执行输出更少，但那些都是被链接器/PDB/库副作用包围的原生构建产物。Roslyn 则在 `artifacts\obj` 下成批创建新 DLL。AV 引擎对新写入的可执行内容和已知平台程序集的处理方式往往大不相同，所以输出的类别和数量同样关键。

**4. 信任形态。** Roslyn 有大量可能已知的 Microsoft/SDK/引用程序集活动——但*同时*也产出大量刚生成的本地输出。Ripgrep 的 Microsoft 输入更少，Rustup/Cargo 用户缓存活动占比更高，刚生成的原生输出虽少但更集中。所以光看 DLL 数量会产生误导：Microsoft 引用 DLL、NuGet 包 DLL、本地新生成 DLL 在 AV 那边走的很可能不是同一条逻辑。

**5. 阶段结构。**

| 阶段 | ripgrep | Roslyn |
| --- | --- | --- |
| 编排 | Cargo crate 图 | dotnet/MSBuild 解决方案图 |
| 编译器核心 | 多个 `rustc.exe` crate 编译 | `dotnet.exe` + `VBCSCompiler.exe` 编译器服务器读取 |
| 元数据压力 | 中等 | 很高 |
| 输出压力 | 集中的原生/PDB/lib/rmeta | 大量托管 DLL/XML/cache/resources |
| 信任特征 | Rustup/Cargo 缓存 + 刚生成的原生输出 | Microsoft SDK/NuGet/引用读取 + 刚生成的 DLL |
| 网络噪声 | 接近本地 | 观察到 restore/cache/类似遥测的 HTTPS |

## AV 产品是怎么踩坑的

现代 AV 早就不是挂在文件读取上做字节匹配的扫描器了。一个典型引擎会混用 minifilter 回调、路径策略、哈希缓存、签名方信誉、云端流行度、脚本规则、行为模型、可执行内容启发式、进程信誉，有时还有专门针对开发工具的放行逻辑。Ripgrep 和 Roslyn 踩中的是这些子系统的不同组合。

具体踩在哪里：

**每次打开和元数据查询的开销。** Roslyn 是更灵敏的探针。某个产品在 `CreateFile`、`QueryOpen`、目录查询或元数据检查上哪怕多加一丁点延迟，Roslyn 就能把这个惩罚放大几百万倍。同一个产品跑 ripgrep 可能看着没事——因为 ripgrep 根本不会产生那种规模的文件系统扇出。

**刚生成的 EXE/DLL 扫描。** 两个工作负载都会触发，但表现不一样。Ripgrep 产出原生可执行文件和链接器产物；Roslyn 产出大量托管 DLL。凡是会深度检查新建可执行文件、等云端信誉返回、或者标记"编译器刚写了一个二进制"的产品，两边都可能被拖慢——只不过 Roslyn 的输出面更广，ripgrep 的原生链接器面更密集。

**签名方和路径信誉。** Roslyn 读取的 Microsoft/SDK/引用内容远比 ripgrep 多。如果一个产品在 Microsoft 发布者/路径信誉上做得好，Roslyn 输入侧的一大块都能走快速通道——但刚生成的输出是全新的本地文件，享受不到这条捷径。Ripgrep 这边 Microsoft 路径的东西更少，Rustup/Cargo 用户缓存的东西更多，所以更依赖哈希/云端流行度，而不是 Authenticode 捷径。

**包缓存信任。** NuGet 和 Cargo/Rustup 缓存归属于不同的信誉体系。NuGet 包可能很常见、也可能有包签名，但里面的 DLL 不一定有 Authenticode 签名。Rustup/Cargo 产物对开发者来说可能稳定常见，但基本不会出现在 Microsoft 允许列表上。不同厂商在这里给出的判断可能天差地别。

**编译器服务器 vs. 短命进程。** Roslyn 的 `VBCSCompiler.exe` 把大量编译器输入读取集中在一个长生命周期进程里。靠进程身份和生命周期来建立信誉的产品，看到的信号跟 ripgrep 那边一堆短命 `rustc.exe` 进程完全不同。

**FAST IO 回退。** Roslyn 的 `FAST IO DISALLOWED` 事件多得多，说明过滤器可能正在把文件系统推向更慢的 IRP 路径。如果某个产品在元数据密集的工作负载上频繁阻止 fast I/O，Roslyn 会把这个问题暴露得一清二楚。

**云端缓存 vs. 本地缓存。** Roslyn 庞大的唯一路径集合会触发远更多的缓存和信誉决策。Ripgrep 则可能由数量更少、但更"可疑"的刚生成产物主导。一个本地哈希缓存做得好、但云端信誉响应慢的产品，可能会出现严重的首次运行惩罚，后续运行则大幅改善。

**行为层的"开发工具"规则。** 编译器天生就会写可执行代码——这正是它们让 AV 头疼的原因。有些产品会给可信工具链开绿灯；有些则把代码生成、链接器输出或未签名二进制当作需要重点关注的行为。Ripgrep 考验的是原生编译器/链接器行为的处理能力，Roslyn 考验的是大规模托管程序集生成的处理能力。

这就是产品排名在两张图之间翻转的原因。一个在 .NET SDK 允许列表和低成本元数据 hook 上做得好的产品，跑 Roslyn 可能成绩不错——直到生成 DLL 阶段开始拖后腿。一个文件打开效率高、但对原生可执行创建盯得很紧的产品，扛得住 Roslyn 的元数据风暴，却可能在 ripgrep 的原生输出上栽跟头。

| `RegQueryValue` | 15,057 | 536,724 | Roslyn 在绝对值上有更多注册表值查询。 |

Roslyn 的 `CreateFile` 约是 ripgrep 的 116 倍，`QueryOpen` 约 496 倍，`QueryNetworkOpenInformationFile` 约 1623 倍。这就是 MSBuild 构建图求值的典型特征：项目文件、导入的 `.props`/`.targets`、目标框架检查、引用程序集、包文件、生成输出和 up-to-date 决策全叠在一起。

反过来看，ripgrep 的 `WriteFile` 事件只比 Roslyn 少约 8.7 倍，但核心事件总数可是少了 63 倍。这就是关键信号：ripgrep *按比例*明显偏写入。这完全符合 Rust/原生构建的特点——编译器/链接器输出、PDB、`.rlib`、`.rmeta`、`.lib`、`.o`、临时导入库和最终二进制，全是写操作。

Roslyn 的跟踪基本把 .NET 生态从头到尾趟了一遍：

- **MSBuild 项目求值和 target 执行：** `.csproj`、`.props`、`.targets`、`.editorconfig`、`.globalconfig`。
- **依赖和包基础设施：** `.nuget`、`.sha512`、包缓存文件、生成的 NuGet props/targets。
- **编译器输入：** `.cs`、`.vb`、`.resx`、引用程序集、分析器 DLL。
- **编译器/构建输出：** `artifacts\obj`、生成的 `.cs`、输出 `.dll`、`.xml`、`.resources`、`.cache`、临时文件。
- **编译器服务器行为：** `VBCSCompiler.exe` 的读取量极大，把编译器输入扫描全集中在一个长生命周期进程里。

这种工作负载对文件元数据缓存差或打开/查询 hook 开销大的产品特别不友好。哪怕读取的字节量并不总是很大，光是唯一路径和探测的数量就已经非常可观了。

网络活动按比例仍然很小，但不是零：14,630 个核心网络事件，主要是发往 Microsoft/Akamai 相关端点的 HTTPS 流量。结合命令和 NuGet/缓存路径来看，可能的原因包括 restore、SDK/工作负载元数据、包/缓存验证，或疑似遥测的流量。如果测试目标是纯本地 AV 开销，未来跑 Roslyn 应该用预先 restore 好的包、加 `--no-restore`、关掉遥测，最好再跑一组断网对照。

## 注册表和配置行为

Ripgrep 的注册表活动按比例更高——占核心事件的 25.2%。热点集中在 `HKLM\System\CurrentControlSet`、`HKLM\SOFTWARE\Microsoft` 和 Visual Studio/MSVC 相关位置，都是经典的 Windows 原生构建动作：编译器发现、链接器设置、Windows SDK 查找、运行时配置、Visual Studio 工具链探测。

Roslyn 的注册表绝对量更大，但占比更小：134 万事件，只占 10.6%。主要集中在 `HKLM\System\CurrentControlSet` 以及 .NET/Visual Studio/SDK 查找。百分比小不代表注册表不重要——只是文件图实在太大，大到注册表退成了背景噪声。

就 AV 性能而言，注册表操作通常不如文件打开和写入敏感。不过注册表密集的设置阶段仍然可能和自我保护 hook、行为监控、策略检查、进程/工具链信誉产生交集。

## 错误和非成功结果

两份跟踪都有大量非成功结果——这对构建系统来说完全正常：

| 结果 | ripgrep | Roslyn | 解读 |
| --- | ---: | ---: | --- |
| `SUCCESS` | 149,553 | 9,374,926 | 已完成操作 |
| `NAME NOT FOUND` | 18,759 | 1,034,436 | 探测可选路径/文件/注册表键 |
| `FAST IO DISALLOWED` | 9,003 | 1,649,094 | Windows 回退到更慢的 IRP 路径；过滤器存在时常见 |
| `REPARSE` | 4,230 | 128,917 | 路径 reparse/符号链接/junction 行为 |
| `NO MORE FILES` | 347 | 70,608 | 目录枚举结束 |

`NAME NOT FOUND` 本身不算失败。构建系统天生就会不断探测可选文件。这一点在 Roslyn 身上特别明显——MSBuild 和 NuGet 会评估大量可能的 import、目标框架、包资产、SDK 文件、生成输出和引用位置，"找不到"是正常流程的一部分。

`FAST IO DISALLOWED` 值得留意：文件系统过滤器会影响 fast I/O 路径能不能走通。Roslyn 的这类事件多得多，所以一个经常把操作推向慢路径的产品，受伤最重的一定是 Roslyn。

## 这些差异怎样影响 AV 性能

### 信任难题

ProcMon 不记录 Authenticode 签名方、catalog 签名、云端流行度或 AV 产品内部的允许列表决策。所以我们没法*证明*某个文件被信任了——分析流水线只能用基于路径的信任/信誉分桶作为近似指标。

信任并不是一边倒的：

| 问题 | 跟踪给出的可能答案 |
| --- | --- |
| 哪个工作负载在绝对值上触达更多可能属于 Microsoft/SDK/引用体系的二进制？ | Roslyn。它在 `C:\Program Files\dotnet`、引用程序集、SDK 位置和 Windows DLL 路径下有更多活动。 |
| 哪个工作负载加载更多平台可信 DLL？ | 绝对数量上是 Roslyn；两者的映像加载主要都是 Windows/SDK 路径。 |
| 哪个工作负载创建更多刚生成的可执行类输出？ | Roslyn 多得多。它生成的 `artifacts\obj` DLL 写入主导类可执行写入。 |
| 哪个工作负载按比例更偏非 Microsoft/用户缓存/工具链形态？ | Ripgrep。它规模更小，但大量活动来自 Rustup/Cargo 用户缓存和刚生成的 `target\release` 输出。 |

增强分析的路径分桶摘要：

| 指标 | ripgrep | Roslyn |
| --- | ---: | ---: |
| Microsoft OS 路径文件事件 | 22,231 | 44,548 |
| Microsoft SDK / Program Files 路径文件事件 | 8,475 | 1,335,173 |
| 用户包/工具链缓存文件事件 | 36,145 Rustup/Cargo | 2,987,591 NuGet |
| 刚生成的构建输出文件事件 | 48,320 | 2,966,226 |
| Microsoft OS/SDK 桶中的类可执行事件 | 21,808 | 1,091,019 |
| 包/工具链缓存桶中的类可执行事件 | 9,795 Rustup/Cargo | 2,587,458 NuGet |
| 刚生成的构建输出中的可执行类事件 | 1,404 | 2,195,529 |
| 刚生成的构建输出中的可执行类写入 | 58 | 216,285 |

实际影响是：Roslyn 巨大的 DLL/引用涉及面并非成本均匀的。很多输入可能是稳定的、已签名的、常见的，或来自包缓存，产品可以很快放行。但 Roslyn 同时也会创建*大量*全新 DLL——这些本地输出享受不到同样的签名/云端/缓存快车道。Ripgrep 这边文件更少、生成的类可执行写入也更少，但 Rustup/Cargo 缓存和刚生成的原生产物大概率不在 Microsoft Authenticode 允许列表的覆盖范围内。

### 超越原始数量：速率和加权压力

光看事件数量，只能知道"AV 有多少次介入机会"，答不了"这些机会可能有多贵"。百分比能看出形态，却把规模藏起来了。所以流水线报告四个层级：

| 层级 | 回答什么问题 | 为什么重要 |
| --- | --- | --- |
| 绝对数量 | AV 暴露面有多大 | Roslyn 有远更多文件打开、元数据查询、失败探测和写入 |
| 百分比 | 这是什么类型的工作负载 | ripgrep 按比例偏写入；Roslyn 偏元数据/打开 |
| 每秒速率 | 事件流有多密集 | Roslyn 虽然运行更久，但每秒事件更多 |
| 加权压力 | 哪些事件可能更贵 | 刚生成的可执行写入和新构建输出比已知 Microsoft OS/SDK 读取权重更高 |

加权压力摘要：

| 指标 | ripgrep | Roslyn |
| --- | ---: | ---: |
| 加权压力分 | 877,977 | 52,713,707 |
| 每秒压力 | 15,369 | 73,520 |
| 每 1,000 事件压力 | 4,347 | 4,175 |
| 最高压力操作组 | write | write |
| 最高压力信任桶 | fresh build output | fresh build output |
| 最高压力阶段 | link/resources | output/write phase |

这个分数是启发式的，不是实测延迟——权重设计是透明的：元数据查询权重低，读取偏低/中，写入更高，映像加载更高，可执行类写入最高。可能属于 Microsoft OS/SDK 的路径用较低乘数；刚生成的构建输出和用户/包/工具链路径用更高乘数。

有了这层加权，结论就不一样了。Roslyn 的总 AV 压力遥遥领先，因为它体量巨大而且创建了大量刚生成的 DLL。但 ripgrep 每 1,000 事件的压力密度反而略*高*——因为它较小的事件流高度集中在写入和刚生成的原生产物上。**Roslyn 是总量上更大的压力测试；ripgrep 是单位密度更高的原生产物压力测试。**

未来运行中值得跟踪的信誉因素：

- Authenticode 签名方和 catalog 签名：Microsoft 签名的 OS/SDK/引用二进制可能比未签名的本地输出走更快的 AV 路径。
- 云端流行度和文件年龄：常见的 SDK/NuGet/Rustup 文件可能已被收录；新构建输出则不在库里。
- 路径和发布者信任：`C:\Windows`、`Program Files\dotnet`、Visual Studio、Windows Kits 这些路径跟 `C:\bench` 或用户目录缓存在 AV 眼里完全不是一回事。
- 生成的类可执行输出：新写入的 `.exe`/`.dll` 可能触发更深层的静态、行为或信誉检查。
- 包缓存信任：NuGet 包可能有包签名、也可能很常见，但里面的 DLL 不一定有 Authenticode 签名。
- 工具链缓存信任：Rustup/Cargo 产物可能稳定常见，但不带 Microsoft 签名。
- 映像加载 vs. 文件读取 vs. 文件写入：AV 产品对"加载 DLL"、"把 DLL 当数据读"和"写入新 DLL"的处理方式可能大不相同。
- 脚本和 MOTW 语义：`.ps1`、`.js`、下载文件以及 Zone.Identifier/MOTW 可能显著改变信誉判定路径。
- 缓存范围：产品的本地缓存、云端缓存、文件哈希缓存和 VM 重置行为可能导致首次入云（first-cloud-seen）与后续平均跑分结果出现明显差异。

### Ripgrep 最容易踩到什么

主要 AV 压力点：

- 扫描新写入的编译器/链接器产物。
- 扫描 PDB、`.lib`、`.rmeta`、`.rlib`、`.o`、`.a` 和最终可执行输出。
- 拦截 `%TEMP%` 下的链接器临时文件。
- 反复读取 Rust 工具链库和 Rust 标准库产物。
- 原生 Windows SDK/MSVC 工具链发现和注册表探测。

可以预期：

- 每个新建的对象/库/调试产物都要扫一遍的产品，影响会很明显。
- 对 Rust 工具链库有良好路径/哈希缓存的产品，缓存热起来之后应该会明显好转。
- 把编译器/链接器输出当成可疑可执行生成行为的产品，即使唯一路径不多，也可能带来明显延迟。

### Roslyn 最容易踩到什么

主要 AV 压力点：

- 横跨源文件、SDK、NuGet、引用程序集、生成文件和输出文件的大量文件打开/查询/读取。
- 大量 `.dll` 读写——分析器、引用程序集、编译器/运行时组件和输出。
- 通过 `.props` 和 `.targets` 执行 MSBuild/NuGet target 与属性求值。
- 通过路径存在性、时间戳、元数据和目录枚举执行增量构建和依赖图检查。
- 编译器服务器进程读取大量编译器输入。

可以预期：

- 每次打开或查询的过滤开销偏高的产品，哪怕字节扫描不多也可能受影响。
- VM 重置后缓存复用能力弱的产品，会反复为 SDK、NuGet、引用程序集和分析器路径付出代价。
- 对 DLL、源生成器、分析器或新生成程序集应用更重规则的产品，影响会很明显。
- 对大量不同 DLL/源文件/生成路径做云端信誉查询的产品，首次入云影响可能很大。

## 速查表

| 维度 | ripgrep | Roslyn |
| --- | --- | --- |
| 生态 | Rust/Cargo/原生 MSVC 链接 | .NET/MSBuild/Roslyn/NuGet |
| 规模 | 紧凑 | 很大 |
| 唯一文件 | 数千级 | 十万级 |
| 主要压力 | 产物写入和链接器/编译器输出 | 文件图遍历、元数据检查、引用、包 |
| 主要进程形态 | `cargo` + `rustc` + `link` | `dotnet` + `VBCSCompiler` |
| AV 缓存问题 | 产品能否高效处理生成的原生产物？ | 产品能否高效处理庞大的托管图遍历和生成 DLL？ |
| 网络敏感性 | 接近本地 | 观察到一些 restore/元数据/类似遥测的 HTTPS 活动 |

这组测试的价值就在于对比。把它们当作“编译时间”的重复样本，等于把真正的信号白白丢掉了。

## 下次我们会怎么做

1. **图表分开出。** Ripgrep 和 Roslyn 值得各有自己的结果——它们的 OS 操作结构差异太大，合并成一个数字会把有意思的产品行为差异抹平。

2. **当子工作负载评分，不当重复样本。** 一个产品在 ripgrep 上表现优秀、在 Roslyn 上表现糟糕，或者反过来，背后都有说得通的技术原因。

3. **加一个严格离线的 Roslyn 变体。** 当前跟踪里有一些网络活动，`dotnet build` 可能隐式 restore 或触碰工作负载/包元数据。如果目标是纯本地 AV 开销，应该用预先 restore 好的包、`--no-restore` 并关掉遥测。

4. **首次跑和缓存预热后分开报。** 首次入云的跑法能捕捉产品冷缓存和云端信誉行为；后续平均值反映的是缓存稳定后的日常开发体验。两者说的是不同的事。

5. **Roslyn 是离群值时，查文件打开/查询延迟。** Roslyn 的缩影受路径和元数据行为主导——不是简单的“写入字节数”模型。

6. **Ripgrep 是离群值时，查产物创建。** 重点看新建的编译器/链接器产物、PDB 写入和临时导入库行为。

## 局限性

ProcMon 跟踪是观察性的——它能显示发生了哪些操作，但没法精确告诉你每个 AV 产品在每个操作上到底加了多少延迟。数量仍然有意义，因为它展示了文件系统过滤器的暴露面；但要做真正的时延归因，还需要每个产品的时序数据或 ETW/WPA 级别的延迟分析。

核心构建进程的过滤基于 ProcMon 的进程创建关系。这比按进程名静态过滤要好，但也不完美：会包含 `Conhost.exe` 这类控制台/辅助子进程，也可能漏掉不是基准测试进程后代、但确实做了构建相关工作的长期系统服务。

生成 JSON 中的字节计数只能当粗略参考值。ProcMon 的 `Detail` 字段因操作类型而异；本分析关注的是操作结构、路径扇出、进程组合、文件类型组合和信誉暴露面——不是精确的字节核算。

## 最后一句话

Ripgrep 是紧凑的原生产物压力测试。Roslyn 是大面积铺开的托管图压力测试。它们都是编译工作负载，但给 AV 出的是完全不同的考题。这正是两个都应该留在测试套件里的原因。
